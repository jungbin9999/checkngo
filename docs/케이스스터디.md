# 체크앤고 (check&go) — 개발 케이스 스터디

> 청년 지원 정책 매칭 서비스. 기획 → 상세설계 → 구현 → 배포까지 AI(Claude Code)와 협업해 완주한 사이드 프로젝트.

- **라이브**: https://checkngo.vercel.app
- **저장소**: https://github.com/jungbin9999/checkngo (public)
- **스택**: Next.js(App Router) · TypeScript · Tailwind CSS · Supabase(Auth + Postgres + RLS) · Vercel
- **역할**: 기획 · 개발 · 배포 (1인)

---

## 1. 문제 정의 (Why)

청년이 정부 지원 정책을 놓치는 진짜 이유는 **"정보가 없어서"가 아니라** 자격 확인·서류 준비·마감 관리가 여러 곳에 흩어져 있어 **그 과정에서 지쳐 포기하기 때문**이다.

- 실제 수급 대상자 조사: 포기 사유 1위가 "기준이 엄격해 신청해도 안 될 것 같아서(자격 오판, 35.4%)" > "절차가 복잡해서(11.9%)" > "제도를 몰라서(5.6%)"
- 즉, 풀어야 할 지점은 **"내 상황에 맞는지 판단"** 과 **"절차 부담"** → 정보 제공이 아니라 *발견에서 신청 준비까지 이어지는 흐름 설계*가 핵심.

→ **프로필 하나로 조건 매칭 → 서류 체크리스트 → 마감 관리를 하나로 잇는다.** MVP 타깃은 1인가구 청년.

## 2. 핵심 기능 (What)

| 기능 | 내용 |
|---|---|
| **조건 매칭** | 나이·취업상태·소득·주거형태·거주지역 5개 조건으로 맞춤 정책 필터링. 적용 조건 칩 표시, 기본순/마감임박순 정렬, 소득 경계/초과 배지, 지역 미지원 "준비중" 처리 |
| **정책 상세** | 자격조건·신청기간·설명 + 서류 체크리스트(진행상황 저장) + 스크랩 + 신청 사이트 바로가기 |
| **마이페이지** | 스크랩 목록, 서류 준비 진행률, 신청완료/삭제, 마감 알림 on/off(개별·전체) |
| **캘린더** | 월별 마감 정책을 달력 + 마감일 순 리스트로 확인, 별 클릭으로 즉시 스크랩 |

## 3. 개발 과정 (How)

**기획 → 상세설계 → 구현 → 배포** 순으로 진행. 상세설계 단계에서 **정책 정의서(필터링/매칭 규칙)를 SSOT(단일 진실 공급원)** 로 문서화하고, 구현은 항상 그 문서를 기준으로 판단하도록 했다.

### 진행 순서
1. Next.js + TypeScript + Tailwind 세팅, Supabase 연동
2. DB 스키마 설계 → 마이그레이션 SQL → 실제 Supabase 적용 (`profiles`/`policies`/`scraps` + RLS)
3. 정책 12건 큐레이션 → 시드 SQL 적재
4. 매칭 로직(`matching.ts`) 구현 + 시뮬레이션 검증
5. 화면 구현: 로그인/회원가입 → 프로필 입력 → 홈(매칭) → 정책 상세 → 마이페이지 → 캘린더
6. 공통 헤더 컴포넌트 추출, 공통 UI(로딩/에러/반응형) 정비
7. GitHub → Vercel 배포

## 4. 주요 의사결정

기술보다 **"무엇을 판단하고 무엇을 위임/제외할지"** 를 관리한 지점들.

- **기준일 고정(`REFERENCE_DATE = 2026-08-15`)**: 포폴 데모라 정책 데이터를 고정. "오늘"을 실제 날짜(`new Date()`) 대신 고정 상수로 판단해, 배포 후 시간이 지나도 마감/D-day/나이 계산이 일관되게 보이도록 함. (안 그러면 시간이 지나며 모든 정책이 "마감"으로 표시되는 버그)
- **소득 매칭의 역설 방지**: 소득이 기준을 초과해도 완전히 제외하지 않고 **경고 배지 + 목록 하단 배치**로 노출. "모름"으로 입력한 사람보다 *정직하게 초과라고 입력한 사람이 불리해지는* 역설을 막기 위함.
- **나이는 저장하지 않고 계산**: 프로필엔 `birth_date`만 저장하고 나이 구간은 매칭 시점에 계산. (나이는 시간이 지나면 바뀌는 값이라 저장하면 stale)
- **RLS(Row Level Security)**: 사용자 데이터(`profiles`/`scraps`)는 본인 행만 접근. 정책(`policies`)은 전체 읽기 공개. 프론트에는 공개용 anon 키만 사용.
- **매칭 로직을 순수 함수로 분리**: `matching.ts`를 UI와 분리해 시뮬레이션으로 규칙 정합성을 먼저 검증한 뒤 화면에 연결.
- **스코프 관리 (MVP 경계)**: 이메일 알림 발송(Resend + Cron)은 외부 연동/비용이 붙어 포폴 범위에서 **의도적으로 제외**하고, "알림 설정이 DB에 저장되는" UI까지만 구현. 만들 것과 안 만들 것을 명확히 구분.

## 5. 트러블슈팅 (배운 점)

실제로 막히고 해결한 지점들 — 포트폴리오에서 "과정"을 보여주는 소재.

- **환경변수 오설정으로 앱 전체 404**: `.env.local`의 `NEXT_PUBLIC_SUPABASE_URL` 값에 `/rest/v1/`가 잘못 붙어 있어 Supabase 클라이언트가 경로를 이중 생성 → 모든 호출 404. 원시 REST 응답까지 파고들어 원인 특정 후 교정. (교훈: 클라이언트 SDK가 base URL에 경로를 덧붙인다는 점)
- **RLS 검증**: 배포 전, anon 키로 다른 사람 데이터가 안 읽히는지 실제로 확인해 RLS가 동작함을 검증.
- **`next/image` 캐싱으로 로고가 안 바뀜**: 같은 `/logo.png` URL을 덮어써도 Next의 이미지 최적화 캐시가 옛 결과를 서빙. **파일명(URL)을 버전으로 바꿔** 캐시를 우회.
- **모바일(360px) 헤더 오버플로우**: 가로로 긴 로고 + 4개 네비 항목이 좁은 화면에서 넘침 → 모바일에서 로고·네비 크기를 줄여 반응형 수정.
- **첫 배포**: GitHub(gh CLI로 저장소 생성/push) → Vercel 대시보드 임포트 → 환경변수 등록 → 배포. 이후 `git push`만 하면 자동 재배포되는 파이프라인 구성.

## 7. 배운 점 / 회고

- **문서(SSOT)를 먼저 확정하니 구현 판단이 흔들리지 않았다.** 매칭 규칙을 정책 정의서에 못 박아두고 코드는 그걸 따르게 하니, 애매한 케이스마다 "문서 기준"으로 결정할 수 있었다.
- **"안 만들 것"을 정하는 게 스코프 관리의 핵심.** 이메일 발송을 과감히 제외해 MVP를 실제 배포 가능한 크기로 유지했다.
- **데모 데이터의 시간 문제**를 기준일 고정으로 해결하며, "배포 후에도 유효한 데모"를 위한 설계를 배웠다.
- **RLS·환경변수·이미지 캐싱** 등 배포/보안의 현실적인 함정들을 직접 부딪히며 익혔다.
- 기획부터 배포까지 **전 과정을 혼자 완주**하며, 각 단계에서 무엇을 직접 결정하고 무엇을 위임/제외할지 판단하는 감각을 얻었다.
