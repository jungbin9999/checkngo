# AI 협업 프로젝트: 체크앤고(Check & Go) — 청년 정책 매칭 서비스

> 상위기획 → 상세기획 → 바이브코딩 제작 → 배포 → 데이터 수집/회고
> 진행하다가 생략하거나 추가해도 괜찮음. 이건 방향을 잃지 않기 위한 체크리스트.
> AI(클로드)와 함께 기획부터 배포까지 전 과정을 진행한 사이드 프로젝트

---

## 서비스명: 체크앤고 (Check & Go)

서류 체크리스트를 확인하고 바로 신청하러 간다는 유저 저니를 이름에 그대로 담음. "Check(자격·서류 확인) → Go(신청하러 이동)"의 2단계 흐름이 서비스명 자체에서 직관적으로 드러남.

---

## 서비스 한 줄 요약 (초안)

내 상황(나이/취업여부/주거형태/소득/지역)을 입력하면, 나에게 맞는 정부 지원 정책만 필터링해서 보여주고, 신청 마감일을 캘린더에 등록하고, 필요한 서류 체크리스트까지 챙겨주는 서비스.

- **MVP 타겟**: 1인가구 청년
- **핵심 차별점**: 정책 "찾기"에서 끝나지 않고 "신청 완료"까지 이어지게 함
- **참고 벤치마킹**: 원티드/잡코리아의 필터링+추천+마감관리 구조를 정책 추천에 적용

---

## 1단계 — 상위기획

문제와 방향을 정의하는 단계. "왜 이 서비스가 필요한가"를 데이터와 논리로 뒷받침.

- [x] **트렌드/니즈 분석**
  - 청년정책 특화 통계는 부족해서, 복지제도 전반의 사각지대 조사 결과로 논리 구성
  - 한국보건사회연구원 조사(지자체 복지담당자 700명): 복지 사각지대 발생 원인 1위 "대상자가 신청하지 않아서"(45.7%), 그 중 공무원들은 "몰라서"(70.9%)를 최다 원인으로 인식
  - 그러나 실제 수급 대상자(기준중위소득 30% 이하) 조사에서는 결과가 다름: "기준이 엄격해 신청해도 안 될 것 같아서"(35.4%, 자격 오판) > "절차가 복잡하고 번거로워서"(11.9%) > "제도를 잘 몰라서"(5.6%)
  - → 시사점: "정보 부족"보다 "내 상황에 맞는지 판단이 안 서는 것"과 "절차 부담"이 실제 포기 원인. 서비스가 풀어야 할 지점과 정확히 일치
  - 참고: 정부의 맞춤형 급여 안내 제도(복지멤버십)조차 2023.8 기준 가입률 19.8%에 그침 → 정보 제공만으로는 부족, 신청 완료까지 이어지는 흐름 설계가 핵심이라는 근거
  - 주변 실사례: 조건 확인 단계에서 막히는 경우 + 확인할 게 많아 미루다 마감 놓치는 경우 (→ 페르소나에 반영)
  - 📊 첨부 이미지: `donut_gov.png`(공무원 인식 조사), `donut_actual.png`(실제 수급대상자 응답) — 여기에 드래그해서 삽입
- [x] **사용자 니즈 분석 & 페르소나**

  **1인가구 미혼 청년 (MVP 타겟) — 서연, 27세, 계약직 사무직**

  - 상황: 서울 원룸 자취 중. 계약직이라 소득이 애매하게 걸쳐 있어 지원 대상 여부 판단이 어려움
  - Problem: "청년월세지원 대상인지 소득 기준표를 봐도 이해가 안 돼서 포기했어요."
  - Needs: "나이·취업상태·소득·주거형태 넷만 넣으면 되는지 바로 알고 싶어요."

  **신혼부부(무자녀) — 지훈(32)·하은(30), 맞벌이 부부**

  - 상황: 결혼 8개월 차. 혼인신고 시점 기준 자격 요건 때문에 타이밍 놓칠까 불안
  - Problem: "부부 합산 소득인지 개별인지, 혼인신고 후 몇 년 이내인지까지 따져야 해요."
  - Needs: "저희 둘 조건을 같이 넣고 한 번에 보고 싶어요."

  **자녀가 있는 가구 — 민경, 35세, 생후 4개월 아기 엄마**

  - 상황: 출산휴가 중. 자녀 월령에 따라 신청 가능한 정책이 계속 바뀌어 기한을 놓칠까 봐 불안
  - Problem: "출산 직후엔 정신없어서 뭘 신청해야 하는지 나중에 알았는데 이미 기한이 지났어요."
  - Needs: "아기 월령 따라 바뀌는 걸 계속 따라가면서 알려줬으면 해요."

  **공통 페인포인트**

  - 자격조건 확인 단계에서 막힘
  - 확인할 게 많아 미루다 마감 놓침

  **세 그룹은 조건이 쌓이는 구조**
  ```
  1인가구      나이·취업여부·소득·주거형태
     ↓ (+배우자 조건, 혼인일)
  신혼부부      1인가구 조건 + 배우자·혼인일
     ↓ (+자녀 수·월령)
  자녀가구      신혼부부 조건 + 자녀 수·월령
  ```

  **MVP를 1인가구로 잡은 이유**

  - 1인가구의 4개 조건은 세 그룹 모두의 공통 기반
  - 여기서 매칭 로직을 먼저 검증하면 신혼부부·자녀가구는 조건만 얹어서 확장 가능
  - 가장 절실한 대상을 미루는 게 아니라, 세 그룹을 지탱할 엔진을 가장 먼저 완성하는 순서
- [x] **벤치마킹 및 분석**

  **왜 이 둘을 비교하는가**

  정책 정보 플랫폼은 이 서비스와 같은 도메인(정책 정보)이지만 "찾기"에서 기능이 끝난다는 한계가 있음. 반대로 구직 플랫폼은 도메인은 다르지만, "내 조건에 맞는 것을 필터링하고, 마감을 관리하고, 실행까지 이어지는" UX 구조가 이 서비스가 지향하는 흐름과 정확히 일치함. 그래서 "무엇을 담을지"는 정책 사이트에서, "어떻게 흐르게 할지"는 구직 사이트에서 가져오는 방식으로 비교함.

  - 정책 정보 플랫폼: 정부24(보조금24), 복지로, 온통청년 등
  - 구직 플랫폼(UX 구조 참고): 원티드, 잡코리아, 자소설닷컴

  | 기능 | 원티드/잡코리아/자소설닷컴 | 정책 정보 플랫폼 |
  |---|---|---|
  | 자격/조건 확인 | 다중 조건을 조합해 한 번에 필터링 | 정책별로 개별 확인해야 함, 내 프로필을 저장해두고 전체를 한 번에 걸러주는 방식은 아님 |
  | 추천 | 프로필 기반 우선순위 추천 | 조건에 맞는 정보를 안내는 하나, 정적인 안내에 가까움 |
  | 마감 관리 | 관심 대상 마감일을 캘린더로 자동 표시 | 정책별 마감일은 공고문에 흩어져 있고, 개인화된 캘린더/리마인더로 묶어 보여주는 곳은 찾기 어려움 |
  | 신청 절차 준비 | "지원하기" 버튼으로 매끄럽게 연결 | 정책마다 신청 창구·필요서류가 제각각이라 사전 준비를 도와주는 창구가 없음 |

  **이 서비스가 채우는 지점**

  자격 확인, 서류 준비, 마감 관리가 정책마다 다른 곳에 흩어져 있어, 사용자는 정책을 하나 발견해도 "이게 나한테 맞나 → 뭘 준비해야 하나 → 언제까지 해야 하나"를 매번 스스로 다시 조사해야 함. 이 서비스는 프로필을 한 번 저장해두면 이 과정을 하나의 흐름(조건 매칭 → 서류 체크리스트 → 마감 캘린더 → 신청 연결)으로 이어지게 함. 원티드가 채용 정보 자체를 새로 만든 게 아니라 필터링·추천·마감관리를 하나로 묶어 승부했듯, 이 서비스도 "정보의 존재"가 아니라 "흩어진 과정의 통합"에서 차별점을 가져감.
- [x] **서비스 컨셉 정의서**

  **한 줄 컨셉**

  자격 확인부터 서류 준비, 마감 관리까지 흩어져 있던 과정을 프로필 하나로 이어주는 정책 매칭 서비스.

  **핵심 기능 3가지**

  - 조건 매칭: 내 상황(나이·취업여부·소득·주거형태)에 맞는 정책만 우선순위로 필터링
  - 서류 체크리스트: 정책별로 필요한 서류를 미리 확인하고 준비 여부 체크
  - 마감 캘린더: 신청 마감이 임박한 정책을 캘린더로 한눈에 확인

  **서비스의 필요성**

  정책마다 자격 확인, 서류 준비, 신청 절차가 제각각인 채 여러 곳에 흩어져 있어, 정책을 하나 발견해도 "나한테 맞는지 → 뭘 준비해야 하는지 → 언제까지 해야 하는지"를 매번 다시 조사해야 한다. 실제로 정책을 놓치는 이유는 정보가 없어서가 아니라, 이 과정이 여러 단계로 나뉘어 있어 중간에 지쳐서 포기하기 때문이다.

  **기대효과**

  - 조건 입력 한 번으로 나에게 맞는 정책을 바로 확인
  - 서류 체크리스트로 신청 직전 좌절 방지
  - 마감 캘린더로 신청 타이밍을 놓치지 않음
- [x] **MVP 스코프 정의서**

  **포함 (Do)**

  - 타겟: 1인가구 미혼 청년
  - 회원가입 + 로그인, 프로필 저장 (나이·취업여부·소득구간·주거형태·거주지역 한 번 입력하면 재사용)
  - 정책 범위: 국가 단위 정책 + 서울 지역 정책 (1인가구 대상 10~20개, 직접 큐레이션)
  - 핵심 기능 3가지: 조건 매칭 필터링 / 서류 체크리스트 / 마감 캘린더
  - 플랫폼: 웹(반응형) 또는 PWA
  - 데이터 소스: 온통청년 Open API + 직접 큐레이션(서류/체크리스트 정보)
  - 신청 연결: 실제 신청 사이트로 외부 링크 연결
  - 리마인더: 이메일 알림 (PWA면 푸시도 가능)

  **제외 (Not now, 향후 확장 계획)**

  - 신혼부부·자녀가구 등 다른 타겟 그룹
  - 서울 외 지역 정책 (다른 지역은 "준비 중" 안내)
  - AI 기반 추천 고도화 (일단 규칙 기반 필터링)
  - 커뮤니티/후기/랭킹 기능
  - 네이티브 앱

## 2단계 — 상세기획

구조와 로직을 설계하는 단계. 실제로 개발에 넘길 수 있는 수준까지 구체화.

- [x] **IA (정보구조도)**

  ```
  체크앤고
  ├── 온보딩
  │   ├── 회원가입/로그인
  │   ├── 프로필 입력 (생년월일 → 나이 자동계산 · 취업여부 · 소득구간 · 주거형태 · 거주지역)
  │   └── 거주지역 미지원 안내 (서울 외 지역 → "준비 중" 화면)
  ├── 홈 (메인)
  │   ├── 나에게 맞는 정책 리스트 (정렬: 마감임박순/기본순)
  │   ├── 초기화 버튼 (프로필 필터링 끄고 전체 정책 보기)
  │   ├── 빈 상태 (조건에 맞는 정책 없음 → 조건 넓히기 안내)
  │   └── 마감 임박 위젯
  ├── 정책 상세
  │   ├── 자격조건 상세
  │   ├── 필요서류 체크리스트
  │   ├── 신청기간/마감일
  │   ├── 스크랩 버튼
  │   └── 신청 사이트 바로가기 (외부 링크)
  ├── 캘린더
  │   └── 필터링된 정책 전체 표시 (기본), 스크랩한 정책은 볼드+컬러로 강조
  ├── 마이페이지
  │   ├── 프로필 수정
  │   ├── 스크랩한 정책
  │   ├── 체크리스트 진행현황
  │   └── 알림 설정 (이메일 리마인더 on/off, 주기)
  ```

  **캘린더 표시 규칙**: 내 조건에 맞는 정책 전체는 기본 텍스트로, 스크랩한 정책은 볼드 처리 + 강조 컬러로 구분 → 후보군과 실제 챙기는 정책을 시각적으로 분리

  **예외 상황 처리**

  - 조건에 맞는 정책이 없을 때: 빈 상태 화면으로 조건 완화 유도
  - 거주지역이 MVP 지원 범위(서울) 밖일 때: 준비 중 안내
  - 리스트가 많을 때: 마감임박순 정렬 기본 제공
- [x] **화면 플로우**

  온보딩부터 신청 사이트 이동까지의 주요 흐름과 분기. 순환 구조(루프)가 있으면 Mermaid 자동배치가 깨지기 때문에 3개로 분리함. 노션 Mermaid 코드블록(언어: Mermaid)에 각각 붙여넣고 "미리보기" 모드로 확인.

  **① 온보딩 & 매칭 흐름 (앱 진입 → 홈)**

  ```
  flowchart TD
      Start([앱 진입]) --> AuthCheck{로그인 상태?}
      AuthCheck -->|비로그인| HomeGuest["홈 - 전체 정책 카드<br/>(→ ②흐름 참고)"]
      AuthCheck -->|로그인| ProfileCheck{프로필 있음?}
      ProfileCheck -->|없음| ProfileInput["프로필 입력<br/>생년월일(→나이 자동계산) · 취업상태 · 소득수준(모름 가능) · 주거형태 · 거주지역"]
      ProfileCheck -->|있음| Matching[조건 매칭 실행]
      ProfileInput --> Matching

      Matching --> IncomeCheck{소득수준 = 모름?}
  ```

  **UX 노트 — "제한없음" 조건과 초기화 버튼**

- [x] **정책 정의서 (필터링/매칭 로직)** (★핵심 기획 문서)

  **1. 필드별 매칭 규칙**

  | 필드 | 사용자 값 | 정책 조건 값 | 매칭 규칙 |
  |---|---|---|---|
  | 나이 | 19~24 / 25~29 / 30~34 / 35~39 중 1개 | 해당 구간 배열 (제한없음 포함) | 정책 조건에 "제한없음" 포함 → 통과 / 아니면 사용자 구간이 배열에 포함되는지 확인 |
  | 취업상태 | 재직자 / 자영업자·프리랜서 / 미취업자 / 기타 중 1개 | 해당 상태 배열 (제한없음 포함) | 위와 동일 방식 |

  | 거주지역 | 시/도 값 | category(국가단위/지역단위) | 국가단위 → 지역 무관 항상 통과 / 지역단위(서울) → 사용자 지역이 서울일 때만 매칭 성공, 아니면 "준비중" 카테고리로 별도 노출 (매칭 실패로 카운트하지 않음) |

  | 사용자 소득 | has_income_condition=true일 때 처리 |
  |---|---|
  | 100만원 미만 | 통과 (안전권) |

  **2. 전체 결합 규칙**

  5개 필드는 모두 **AND**로 동시에 만족해야 매칭 성공. 단, 정책 조건이 "제한없음"이거나 해당 조건 자체가 없는 필드(예: has_income_condition=false)는 자동 통과로 간주하고 AND 판단에서 실질적으로 제외됨.

  - **제한없음 단독값 규칙**: 데이터 큐레이션 시 "제한없음"은 항상 단독 값으로만 입력 (다른 구체값과 혼합 입력 금지). 로직 안전장치로, 혹시 배열에 "제한없음"과 다른 값이 함께 있어도 "제한없음"을 우선 적용해 자동 통과 처리
  - **마감 여부는 매칭과 무관**: 마감 여부(deadline 경과 여부)는 AND 매칭 판단에 포함되지 않는 순수 디스플레이 속성. 매칭 성공 여부와 별개로 정렬·반투명 처리에만 사용

  **3. 예외 처리 규칙**

  - 거주지역: 국가단위 정책은 지역 무관 매칭, 지역단위(서울) 정책은 서울 거주자만 매칭 성공 처리하고 그 외 지역은 "준비중" 카테고리로 별도 노출. **단, 지역 조건 외 나머지(나이/취업상태/소득/주거형태)가 모두 일치할 때만 "준비중" 노출, 다른 조건까지 불일치하면 완전 미노출**
  - 소득 "모름": 소득조건 있는 정책도 걸러내지 않고 직접확인 배지로 노출
  - 나이: 생년월일 기반 계산 나이가 19~39 범위 밖이면 매칭 결과 0건 → 빈 상태 화면
  - 초기화 버튼 클릭: 위 모든 필터링 규칙을 무시하고 전체 정책 노출
  - **상시형 정책**: 마감일이 없어 D-day·마감임박 표시를 하지 않음. 캘린더에 등록하지 않고, 스크랩해도 마감 임박 알림이 발송되지 않음 (알림 설정 화면에 "이 정책은 상시 접수라 마감 알림이 없어요" 안내)
  - **회차마감형 정책이 마감된 후 다음 회차가 열리면**: 기존 마감 카드를 되살리지 않고 새 policy_id로 신규 카드 발행 (구직 공고 재등록과 동일한 방식). 기존 마감 카드는 반투명 처리 유지하되 "다음 회차 보기 →" 링크로 같은 policy_group_id의 최신 진행중 회차로 연결. 다음 회차가 아직 없으면 "다음 회차 미정"으로 표시
  - **기준일 고정(freeze)**: 데이터 수집 마감(2026-08-15) 이후로는 정책 데이터를 계속 갱신하지 않으므로, 마감 여부·마감임박 정렬·캘린더 표시에 쓰이는 "오늘 날짜"를 실제 시스템 날짜 대신 **고정된 기준일(2026-08-15)**로 처리. 배포 링크로 유입되는 시점이 실제로 언제든 서비스 내부적으로는 항상 "8/15 기준"으로 보여 마감 정책만 쌓이는 현상을 방지. 단, 신청 사이트 바로가기는 외부 실제 사이트로 연결되므로 이 고정과 무관하게 실제로는 마감된 페이지가 뜰 수 있음 → 화면에 "2026년 8월 15일 데이터 기준의 포트폴리오 데모" 안내 문구 노출 필요 (배지 없이 원본 그대로)

  **취업상태 원본 데이터 → 표준값 매핑표** (온통청년 등 API 원본은 더 세분화되어 있어 큐레이션 시 아래 기준으로 통일)

  **4. 정렬 규칙**

  - 마감임박순: deadline 오름차순 정렬. 이미 마감된 정책은 정렬 순위와 무관하게 리스트 최하단에 반투명 처리로 고정 배치. 상시형(deadline_type=상시)은 D-day 개념이 없어 마감임박순 정렬 대상에서 제외하고 별도 그룹("상시 접수")으로 하단에 모아 표시

  **📌 이 문서를 보는 법 — 기획자와 AI의 역할 구분**

  API 명세는 기획자가 100% 이해하고 있어야 하는 영역은 아님. 이 문서에서 기획자가 직접 판단한 부분과, 바이브 코딩 단계에서 AI(Claude Code)가 실제 API를 호출해보며 채워야 할 부분을 의도적으로 구분해서 작성함.

  - **기획자가 판단한 부분(구조적 결정)**: API 키를 백엔드에서만 써야 한다는 보안 원칙, 실시간 호출 대신 배치 동기화로 가야 하는 이유, 필요서류처럼 API에 없을 데이터를 수동 큐레이션으로 보완하는 전략, 발생 가능한 오류 케이스를 미리 예측해 대비해두는 것 — 이런 건 기술 지식보다 "리스크를 예측하고 우선순위를 정하는" 기획자의 판단 영역
  - **AI가 확인할 부분(구현 디테일)**: 정확한 응답 필드명, 인증키 발급 경로의 세부 차이, 정확한 트래픽 제한 수치 — 이런 건 실제로 API를 호출해봐야 알 수 있는 것들이라 "다음 단계에서 확인 필요"로 명시해두고 구현 단계로 넘김

  **Out of scope (이번엔 하지 않는 것)**

  - 신혼부부·자녀가구 등 1인가구 외 타깃 그룹
  - 서울 외 지역 정책
  - AI 기반 추천 고도화 (규칙 기반 필터링만 사용)
  - 커뮤니티·후기·랭킹 기능
  - 네이티브 앱

  **우선순위 등급**

  | 등급 | 기능 | 의미 |
  |---|---|---|
  | P0 | 회원가입/로그인·프로필 저장, 조건 매칭 필터링, 정책 상세(자격조건·신청기간·신청사이트 연결) | 이게 없으면 서비스가 성립 안 됨 |
  | P1 | 서류 체크리스트, 스크랩, 마감 캘린더, 이메일 알림(7일전/1일전) | 이게 있어야 "체크앤고"라는 정체성이 성립 (핵심 차별점) |
  | P2 | 초기화 버튼, 정렬 옵션, 마이페이지 세부 관리(신청완료 배지·개별 알림), 예외 화면 고도화 | 있으면 좋지만 초기 배포 후 추가해도 무방 |

  → P0+P1까지가 실질적 MVP. 시간이 부족하면 P2는 배포 이후로 미뤄도 서비스 정체성은 유지됨.

  **일정/마일스톤** (역산 기준: 배포+데이터수집 완료 목표 8월 중순)

  | 단계 | 내용 | 목표 시점 |
  |---|---|---|
  | 상위기획 | 트렌드·페르소나·벤치마킹·컨셉·스코프 | 완료 |
  | 상세기획 | IA·화면플로우·데이터구조·정책정의서·정책큐레이션·API명세·PRD | 완료 |
  | 바이브코딩 제작 | P0 → P1 순으로 구현, P2는 시간 남으면 진행 | 배포 1~2주 전까지 |
  | 배포 | 실서비스 오픈, 데이터 기준일 고정 준비 | 8월 중순 |
  | **데이터 수집 마감·기준일 고정** | 정책 데이터 갱신 중단, "오늘" 기준을 2026-08-15로 고정 | **2026-08-15** |
  | 데이터 분석·회고 | 성공지표 확인, 정성 피드백 정리, 케이스 스터디 작성 | 배포 후 1~2주 |

  → 8월 15일 이후로는 정책 데이터가 정적으로 멈춘 "포트폴리오 데모" 상태로 운영. 화면에 데이터 기준일 안내 문구 노출.

  신규 서비스라 기존 앱과의 A/B 비교나 %기반 목표가 무의미함 (배포 규모가 지인·커뮤니티 공유 수준이라 표본이 작음). 그래서 정량 지표는 "목표 달성 여부"보다 **"어디서 막히는지 관찰하는 도구"**로 사용하고, 절대 숫자 목표(참고용)와 정성 피드백을 함께 봄.

  | 지표 | 측정 방법 | 목표치(참고, 배포 20명 기준) | 의미 |
  |---|---|---|---|
  | 가입자 수 | 회원가입 완료 인원 | 20명 이상 | 배포 범위가 최소 관찰 규모에 도달했는지 |
  | 온보딩 완료 인원 | 프로필 입력까지 완료한 인원 | 16명 이상 (80%) | 입력 항목(5개)이 부담스럽지 않은지 |
  | 핵심 퍼널 도달 인원 | 정책상세 → 체크리스트 확인까지 도달한 인원 | 8명 이상 | 서비스 핵심 가치(발견→준비)가 실제로 체감되는지 |
  | 스크랩 인원 | 1개 이상 정책을 스크랩한 인원 | 10명 이상 | 매칭 결과가 실제로 관심을 끌었는지 |
  | 신청 사이트 클릭 인원 | 신청 사이트로 실제 이동한 인원 | 5명 이상 | 최종 행동 전환, 우리가 측정 가능한 마지막 지점 |
  | 정성 피드백 확보 인원 | 짧은 인터뷰·설문에 응답한 인원 | 5명 이상 | 숫자 뒤의 "왜"를 설명해줄 표본 (작은 규모에서는 정량보다 신뢰도 높음) |

  **이탈 지점 관찰**: 위 표만으로는 "왜 안 썼는지"가 안 보이므로, 각 단계 사이(온보딩→매칭결과, 매칭결과→정책상세 등)에서 다음 단계로 안 넘어간 사용자가 있으면 그 지점을 별도로 기록. 정성 피드백에서 우선적으로 질문할 포인트로 활용.

  **리스크 및 가정**

  | 리스크 | 내용 | 대응 |
  |---|---|---|
  | API 승인 지연 | 온통청년 인증키 승인이 즉시가 아닌 수일 소요될 수 있음 | 인증키는 상세기획 초반에 미리 신청, 승인 대기 중엔 MVP 정책 큐레이션(수동 리서치)으로 병행 진행 |
  | 소득 매칭 정확도 한계 | 소득구간을 실제 기준중위소득 %로 정밀 환산하지 않고 근사치(200만원/300만원 커트라인)로 판정 | 정책 상세 화면에 원문 기준을 그대로 노출해 사용자가 최종 판단하도록 보완 |
  | 정책 데이터 신선도 | 실제 리서치 결과 10개 중 6개가 리서치 시점 기준 이미 마감 상태였음(→ 청년정책은 정보 유효기간이 짧다는 걸 데이터로 확인) | 데이터 수집 마감(8/15) 이후 기준일 고정(freeze)으로 대응, 상시형·회차마감형 구분 로직으로 최소한의 최신성 유지 |
  | 소규모 배포로 인한 통계적 한계 | 지인·커뮤니티 공유 수준(약 20명)이라 %기반 지표의 통계적 신뢰도가 낮음 | 정량 지표는 절대 숫자로 관찰하고, 정성 피드백(인터뷰)을 주 판단 근거로 병행 |
  | 1인 기획+개발 리소스 제약 | 상세기획 7개 문서 + 바이브코딩을 혼자 진행 | P0→P1 우선순위대로 진행하고 P2는 배포 이후로 유연하게 조정 |

  **관련 문서 목록** (노션 서브페이지로 연결)

  1. 정책 정의서 (필터링/매칭 로직)
  2. IA (정보구조도)
  3. 화면 플로우
  4. 데이터 구조 설계
  5. MVP 정책 큐레이션 리스트 (스프레드시트)
  6. API 명세 (온통청년 Open API)
