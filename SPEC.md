# 체크앤고 (Check & Go) — SPEC

Claude Code가 세션 내내 참고하는 마스터 스펙. CLAUDE.md보다 상위 개념 — "무엇을 왜 만들고, 뭐가 되면 완료인지"를 담음. 프로젝트 진행하며 계속 업데이트.

---

## 1. 개요

청년 정책 매칭 서비스. 자격 확인부터 서류 준비, 마감 관리까지 흩어져 있던 과정을 프로필 하나로 이어주는 서비스. MVP 타겟은 1인가구 미혼 청년. (자세한 배경: `docs/프로젝트_로드맵.md` PRD의 "배경/목적", "타겟·문제 정의")

---

## 2. 기능 요구사항 (완료 조건 포함)

### 2-1. 로그인/회원가입 (P0) — ✅ 구현 완료

- 이메일/비밀번호 방식만 지원 (소셜로그인 없음)
- 로그인 / 회원가입 탭 전환 가능
- 이메일 형식이 아니면 에러 메시지 표시
- 비밀번호 길이가 Supabase 최소 기준 미달이면 에러 메시지 표시
- 로그인 실패(이메일/비밀번호 불일치) 시 "이메일 또는 비밀번호가 일치하지 않습니다" 표시
- 회원가입 성공 시 자동 로그인 (Confirm email 비활성화 상태)
- 로그인 성공 시 현재는 성공 메시지만 표시 (홈 화면 완성되면 `/`로 리다이렉트하도록 변경 예정)

**완료 조건**: 회원가입 → 자동 로그인 → 로그아웃 → 재로그인이 에러 없이 동작. 이메일 형식 오류·비밀번호 오류·로그인 실패 3개 에러 케이스가 화면에 노출됨.

### 2-2. 프로필 입력 (P0) — 다음 작업

- 생년월일(date picker), 취업상태(단일선택), 소득수준(단일선택, "모름" 포함), 주거형태(단일선택), 거주지역(시/도 선택) 입력
- 5개 필드 모두 필수 입력 (미입력 시 저장 불가, 어떤 필드가 비었는지 표시)
- 저장 시 `profiles` 테이블에 upsert (이미 프로필 있으면 수정, 없으면 신규 생성)
- 생년월일로 나이를 직접 계산해서 저장하지 않음 — `birth_date`만 저장, 나이 구간은 매칭 시점에 계산 (정책 정의서 규칙)
- **거주지역은 전체 시/도를 다 선택 가능** — 서울 외 지역을 선택해도 온보딩이 막히지 않음. 국가 단위 정책은 지역 무관 정상 매칭되고, 지역 단위(서울) 정책만 결과 화면에서 "준비중"으로 표시됨 (정책 정의서 예외 처리 규칙). ⚠️ IA 문서의 "거주지역 확인 → 서울 외 지역 준비중 안내"는 이 로직이 확정되기 전 옛 버전이 그대로 남은 것 — IA 업데이트 필요, 구현은 정책 정의서 기준을 따를 것
- **재방문 시(이미 프로필 있음)**: 화면 진입 시 기존 저장값으로 폼이 미리 채워진 상태로 열림 (빈 폼 아님)

**완료 조건**: 5개 필드 다 입력하고 저장하면 Supabase `profiles` 테이블에 실제 행이 생성/갱신됨. 필수값 누락 시 저장 버튼이 막히거나 에러 표시. 기존 프로필이 있는 사용자가 재진입하면 저장된 값이 폼에 채워져 있음.

### 2-3. 홈 - 조건 매칭 필터링 (P0) — 다음 작업

- 로그인 + 프로필 있는 사용자: 매칭 로직(`matching.ts`) 실행 결과를 카드 리스트로 표시
- 비로그인 또는 프로필 없는 사용자: 전체 정책 카드 표시 (매칭 없이)
- **적용된 필터 조건을 칩(chip) 형태로 시각적으로 표시** — "내 맞춤만/전체보기" 토글만으로는 어떤 조건으로 걸러졌는지 안 보임. 정책 리스트 상단에 현재 프로필 기준 조건(나이 구간·취업상태·소득수준·주거형태·거주지역)을 칩으로 나열
  - MVP 범위: 칩은 현재 적용된 조건을 보여주는 용도(읽기 전용). 칩 또는 옆의 "필터 수정" 링크 클릭 시 프로필 수정 화면으로 이동해서 값 변경 (칩 자체를 눌러 그 자리에서 드롭다운으로 바로 수정하는 인라인 편집은 범위 밖 — 프로필 수정 화면과 로직이 중복되는 걸 피하기 위함, 추후 필요시 확장)
  - "전체보기(초기화)" 상태일 때는 칩 영역이 숨겨지거나 "필터 없음" 표시
- 정렬 기본값: 기본순. **마감 여부가 1차 기준** — 마감된 정책은 배지 상태와 무관하게 항상 최하단. 마감 안 된 것들끼리는 무배지 → 배지 → 소득초과 강한경고 → 준비중 순
- 정렬 옵션: 마감임박순 토글 가능 (진행중 회차마감 deadline 오름차순 → 상시 접수 → 마감됨 최하단)
- 마감된 정책: 카드 반투명 처리, **기본순·마감임박순 둘 다 하단 고정** (정렬 기준과 무관하게 항상 마지막)
- 매칭 결과 0건: 빈 상태 화면("조건 넓히기" 안내)
- 초기화 버튼: 클릭 시 프로필 필터링 무시하고 전체 정책 노출

**완료 조건**: 정책 정의서의 "예시 시뮬레이션" 표와 동일한 입력값을 넣었을 때 동일한 결과(매칭/준비중/미노출)가 화면에 나타남. (matching.ts 유닛 검증은 이미 완료 — 화면에서도 같은 결과인지 재확인) "내 맞춤만" 상태에서 현재 적용된 5개 조건이 칩으로 화면에 보이고, 클릭 시 프로필 수정 화면으로 이동함.

### 2-4. 정책 상세 (P0) — 다음 작업

- 정책명, 자격조건(나이/취업상태/소득/주거형태), 신청기간·마감일, 정책 설명(description) 표시
- 서류 체크리스트: 로그인 상태면 항목별 체크 가능(→ `scraps.checklist_status`에 저장), 비로그인이면 목록만 보임(체크 불가)
- 스크랩 버튼: 비로그인 시 클릭하면 로그인 유도, 로그인 상태면 스크랩 on/off 토글 (`scraps` 테이블에 저장/삭제)
- 신청 사이트 바로가기: 새 탭으로 `apply_url` 열기

**완료 조건**: 정책 상세 진입 시 해당 정책의 모든 필드가 정확히 표시됨. 로그인 상태에서 스크랩 토글이 DB에 반영됨(새로고침해도 유지). 신청 사이트 버튼이 실제 URL로 정상 연결됨.

### 2-5. 공통 UI 상태 규칙 (모든 화면 공통)

- **로딩 중**: 데이터 요청 중엔 스피너 또는 스켈레톤 UI 표시 (빈 화면 방치 금지)
- **에러 발생**: 사용자에게 이해 가능한 문구로 표시 ("문제가 발생했어요, 다시 시도해주세요" 등). 콘솔 에러 메시지를 그대로 화면에 노출하지 않음
- **반응형**: 모바일 화면(360px 이상)에서도 레이아웃이 깨지지 않아야 함 (MVP 스코프 "플랫폼: 웹(반응형)" 기준). 데스크탑 전용으로 만들지 않음

**완료 조건**: 네트워크를 느리게 시뮬레이션했을 때 로딩 표시가 보임. 의도적으로 에러를 발생시켰을 때 사용자 친화적 메시지가 뜸. 모바일 너비에서 주요 화면(로그인, 홈, 정책상세)이 깨지지 않고 보임.

## 3. 비즈니스 로직

상세 규칙(필드별 매칭 규칙, AND 결합, 소득 근사 규칙, 예외 처리, 정렬 규칙)은 `docs/프로젝트_로드맵.md`의 "정책 정의서 (필터링/매칭 로직)" 섹션이 SSOT(Single Source of Truth). 코드 구현 시 반드시 그 섹션 기준으로 판단.

## 4. 데이터 모델

`profiles` / `policies` / `scraps` 3개 테이블. 필드 정의는 `docs/프로젝트_로드맵.md`의 "데이터 구조 설계" 섹션 참고. 실제 스키마는 `supabase/migrations/`에 SQL로 반영되어 있음.

## 5. 기술 스택 · 실행 규칙

스택: Next.js + TypeScript + Tailwind + Supabase + Resend + Vercel Cron + react-day-picker (자세한 건 CLAUDE.md)

커밋 전 확인: `npx tsc --noEmit` / `npm run lint` / `npm run build` 모두 통과해야 함

---

## 6. 경계 설정

**항상 해도 됨**
- 파일 읽기, 코드 조회, `npm run dev`로 로컬 서버 실행
- 단순 조회성 명령(git status, ls 등)

**물어보고 진행**
- 3개 이상 파일에 걸친 스키마/구조 변경 (Plan Mode 사용 권장)
- DB에 실제 데이터 넣거나 바꾸는 작업 (SQL 파일 먼저 만들고 검토 후 실행)
- 새 패키지 설치

**절대 하면 안 됨**
- Supabase 서비스 키(service_role/secret)를 프론트엔드 코드나 `NEXT_PUBLIC_` 환경변수에 넣기
- `new Date()`로 마감/기준일 판단 (반드시 고정된 `REFERENCE_DATE` 사용 — CLAUDE.md 참고)
- 실제 DB에 대한 파괴적 명령(테이블 삭제 등)을 확인 없이 실행
